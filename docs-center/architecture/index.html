CCG Architecture Assessment

Migration plan from the playable local race into a deterministic, replayable, federated card-game platform. This document precedes card zones, cryptographic commitments, proof systems, and consensus work.

Current Repository

  • The React application renders one mature race engine plus heuristic AI, audio, and a Race Vector display.
  • src/engine/ remains UI-independent and stores serializable state.
  • GameAction is shared by UI and AI, but its injected function-valued dice rollers are not serializable commands.
  • Readable event log entries are stored in state, but they are not canonical protocol events and have no sequence or hash chain.
  • The new Portal separates Gateway, Hub, Race, and Account scenes without moving account logic into the engine.
  • src/shared/ now contains reusable account contracts plus canonical serialization and hashing foundations.

Current Turn Loop

TURN_START
  -> resonance roll
  -> standard Collatz pulse step
  -> optional prime capture
  -> OPTIONAL_ACTIONS
  -> combat choice or forced turn end
  -> resonance retention
  -> victory check
  -> NEXT_PLAYER

Pending capture and combat choices are explicit state. This is a strong base for deterministic commands, but random results must become command data or derived protocol input rather than injected runtime functions.

Nondeterminism Audit

  • src/engine/dice.ts: default d6() calls Math.random().
  • src/engine/game.ts: turn and combat actions use that default when no roller is injected.
  • src/ui/Jukebox.tsx: shuffle uses Math.random(); this is presentation-only and outside match validation.
  • Mock account sessions are deterministic local identifiers and explicitly provide no authentication security.

First Deterministic Refactor

  1. Define a versioned protocol command containing actor, match, turn, sequence, previous hash, public parameters, and explicit random input references.
  2. Separate local UI actions from serializable match commands; remove function values from the protocol boundary.
  3. Return typed domain events from validation rather than treating display strings as authoritative events.
  4. Add canonical serialization with sorted object keys, integer-only match fields, UTF-8 rules, explicit nulls, and protocol domain separation.
  5. Add SHA-256 state/event hashing behind an adapter and replay a command transcript from genesis state.
  6. Reject stale hashes, duplicate sequence numbers, wrong actors, and missing deterministic random inputs.

Per-Screen Privacy Model

  • An authenticated account session maps to one match participant and seat credential.
  • A player screen receives public match state plus only that participant's authorized private-zone view.
  • A spectator screen receives public state and proof results but no private hand, deck order, or Snare data.
  • Opponent focus panels are projections of public state, not copies of another player's client state.
  • Local hot-seat mode may reveal all seats for development, but it cannot represent competitive privacy.

Security Limitations

Current account sessions are local development conveniences. They have no password verification, signatures, ownership attestations, remote revocation, collection commitments, or server trust. They must never be treated as competitive credentials.

Hosted Race: Minimum Viable Path

The current all-public race can be hosted before cards, hidden zones, consensus validators, or zero-knowledge proofs. The first online version should use one authoritative service rather than attempting federation early.

Immediate Checkpoint

Local gameplay, heuristic rivals, training-log export, canonical serialization, and hashing foundations are in place. The next implementation milestone is a deterministic command/event/replay layer with reproducible final checksums. Room hosting begins only after replay fidelity is proven.
